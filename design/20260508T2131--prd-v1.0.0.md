## Problem statement

LineagesIO no longer fits cleanly inside a single authority artifact.

The original design corpus still carries the early phase-1 objectives:
FileIO-centered user syntax, broader phase scope, missing live companion
briefs, and consumer-package aspirations that predate the later public-surface
rearchitecture. The later workflow PRDs and tranche decisions, however, now
ratify a different live package-owned architecture:

- `read_lineages(...)` is the first-class package-owned public surface
- `BuilderDescriptor` is the first-class typed builder descriptor
- `FileIO.load(...)` remains a compatibility wrapper
- `load_alife_table(...)` remains the in-memory Tables.jl convenience wrapper
- authoritative tables remain the core ownership boundary
- consumer extensions sit on top of that typed core

The repository is already mostly aligned with that final vision. The code,
README, docs, tests, and examples center `read_lineages(...)`, and the
extension modules now project from a package-owned typed owner rather than from
an ad hoc wrapper-only surface. But the authority story is still fragmented and
one audit finding remains unresolved:

- the old design authority chain cites missing live files
- several legacy objectives were never explicitly classified as superseded,
  deferred, or still required
- the MetaGraphsNext supplied-instance later-edge failure path still needs
  owner classification as either a strong unchanged-and-retryable contract or a
  narrower accepted partial-state contract

This PRD exists to reverse-engineer the final package-owned architecture from
the live repository and the ratified later workflow, while honestly carrying
forward unresolved review gates instead of pretending they were already settled.

## User stories

1. As a package user, I can call `read_lineages("tree.nwk")` and receive the
   package-owned rooted lineage graph load flow without going through a host
   wrapper first.
2. As a package user, I can call `read_lineages("tree.txt"; format = :newick)`
   or `read_lineages(io; source_path = "tree.nwk")` and get an honest explicit
   format contract.
3. As a package user, I can call `read_lineages("tree.nwk", NodeT)` and use a
   library-created basenode materialization path when that target contract is
   supported.
4. As a package user, I can call `read_lineages("tree.nwk", basenode)` and bind
   into a caller-owned basenode or graph target when that supplied-instance
   path is supported.
5. As a package user, I can call
   `read_lineages("tree.nwk", BuilderDescriptor(builder, HandleT[, ParentCollectionT]))`
   and get a first-class typed builder boundary that rejects erased handle
   contracts early.
6. As a compatibility user, I can keep using `load(...)` and the FileIO
   surfaces, but I can also tell that they are compatibility wrappers rather
   than the package-owned public owner.
7. As an in-memory tables user, I can call `load_alife_table(table)` and get
   the convenience wrapper story without confusing it for the first-class file
   or stream load verb.
8. As a user of any load path, I still receive authoritative `node_table` and
   `edge_table` results with retained source annotations preserved under their
   source field names.
9. As a MetaGraphsNext user, I can request the supported library-created
   `MetaGraph` path through `read_lineages(src, MetaGraph)` and either get the
   exact supported target family or an honest rejection.
10. As a MetaGraphsNext user with my own empty target, I can use the
    caller-owned supplied-instance path and know exactly what custom metadata
    behavior and failure atomicity the package actually supports.
11. As a PhyloNetworks user, I can call `read_lineages(path, HybridNetwork)` or
    `read_lineages(path, HybridNetwork())` and have docs and rejection paths
    that match the same package-owned public story.
12. As a downstream consumer, I can still traverse a MetaGraphsNext-backed
    materialized graph through the `AbstractTrees` wrapper without moving
    traversal semantics into core.
13. As a maintainer, I can inspect `src/load_owner.jl` and see the canonical
    package-owned typed owner rather than a wrapper-only orchestration layer.
14. As a reviewer, I can verify contract-sensitive behaviors directly at
    `canonical_load(...)`, `read_lineages(...)`, and the retained wrappers
    rather than inferring correctness from a green suite alone.
15. As a maintainer, I can tell which original design objectives are still live
    in v1.0.0 and which ones are deferred or historical, instead of assuming
    everything in the early phase-1 brief is still a current acceptance gate.
16. As a maintainer, I can read one PRD and understand the current module
    owners, public surfaces, compatibility boundaries, and remaining review
    gates.
17. As a future workflow agent, I can start from this PRD and know when to
    escalate conflict between old design objectives and the later package-owned
    public-surface architecture.
18. As a user affected by the MetaGraphsNext later-edge follow-up, I either get
    a strong unchanged-and-retryable supplied-instance contract or an explicitly
    documented narrower contract with direct tests, but not an ambiguous middle
    state.

## Authorized disruption boundary

- internal redesign allowed: targeted continuation of the current typed
  package-owned owner architecture; extension-owner refactors needed to keep
  contract honesty; docs, tests, and examples updates needed to preserve truth-
  boundary alignment
- internal redesign forbidden: reopening the public names `read_lineages` and
  `BuilderDescriptor`; reclassifying `FileIO.load(...)` away from
  `compatibility wrapper`; reclassifying `load_alife_table(...)` away from
  `convenience wrapper`; weakening authoritative tables or stable materialized
  graph or basenode result order
- external breaking changes allowed: only targeted contract-tightening needed
  to make accepted extension target requests, supplied-instance failure
  behavior, or rejection paths honest, with synchronized tests and docs in the
  same change family
- external breaking changes forbidden: broad migration policy changes,
  rename/removal churn, new public-surface families, or silent narrowing of
  supported contracts
- required migration or compatibility obligations: retain `FileIO.load(...)`
  support; keep wrapper parity where the same semantic remains available; keep
  README, docs, tests, and examples synchronized when a user-visible contract
  changes
- non-negotiable protections: separate lock closure for each surviving review
  finding or compatibility boundary; no green-suite-only closure when a direct
  contract proof exists

## Target architecture

- `read_lineages(...)` remains the single first-class package-owned file or
  stream verb for rooted lineage graph loading.
- `BuilderDescriptor` remains the single first-class typed builder descriptor
  for the package-owned public path.
- `FileIO.load(...)` remains above the package-owned owner as a compatibility
  wrapper, delegating into the same internal typed semantics rather than owning
  a separate truth boundary.
- `load_alife_table(...)` remains above the package-owned owner as the in-
  memory Tables.jl convenience wrapper.
- `canonical_load(...)` remains the canonical typed owner, with source
  descriptors and typed load requests enforcing the true architecture under all
  supported public surfaces.
- Authoritative tables remain central. Format modules preserve structure and
  retained annotations there first, and extension modules project into native
  consumer graphs from that package-owned core.
- Extension owners remain thin and honest. They may interpret retained
  annotations into consumer-native graph objects, but they do not become the
  primary public surface or bypass authoritative tables.
- Multi-surface semantics move together. When a contract changes at an owner,
  the corresponding `canonical_load(...)`, `read_lineages(...)`, retained
  wrapper, docs, example, and rejection-path surfaces move together or the
  deferral is called out explicitly.

## Implementation decisions

- Preserve the later ratified package-owned public-surface boundary:
  `read_lineages(...)` and `BuilderDescriptor` stay fixed.
- Preserve the compatibility and convenience classifications:
  `FileIO.load(...)` stays a compatibility wrapper and `load_alife_table(...)`
  stays the in-memory Tables.jl convenience wrapper.
- Preserve the authoritative-table-first invariant and row-reference delivery
  boundary. Core does not become the semantic owner of format-specific or
  consumer-specific annotation interpretation.
- Preserve exact package-owned typed boundaries. No first-class public typed
  path may reintroduce hot-path `Any`, erased handle collections, or wrapper-
  only runtime recovery as if that were the package-owned owner.
- Treat `MetaGraphsNext`, `PhyloNetworks`, and the MetaGraphsNext
  `AbstractTrees` wrapper as the current live extension support set for v1.0.0.
- Treat the current live docs, examples, and tests as the public truth
  boundary to keep in sync with code. Do not allow workflow or error-path
  drift to survive behind a green suite.
- Do not silently satisfy the missing legacy `brief.md` and
  `brief--user-stories.md` citations. Either restore them, retire them, or
  classify the carried-forward objectives explicitly in later governance work.
- Classify original phase-1 objectives that were not carried into the live
  `read_lineages(...)` architecture as deferred or superseded for v1.0.0 unless
  explicitly re-ratified. This includes, at minimum, earlier emphasis on
  `loadfirst`, `loadone`, `LineageNetwork`, `LineageGraphML`, `PhyloExt`, and
  Aqua/JET closure as immediate gating deliverables.
- The MetaGraphsNext later-edge supplied-instance contract remains an explicit
  review gate until the follow-up tasking under
  `.workflow-docs/20260508T2131--rc1-audit2/` closes it honestly.

  wrapper-only builder normalization
- interface: wrapper parity for supported semantics without becoming the

- responsibility: package-owned parsing, source normalization, and
  authoritative table population for the current live formats
- interface: format-specific source descriptors and authoritative-table inputs
- tested: direct format tests, public-surface tests, and shipped examples

- responsibility: core structural types, row references, construction protocol,
  authoritative tables, and post-load views
- interface: stable authoritative table and construction protocol boundary
- tested: core owner tests and extension-path integration tests

### `ext/MetaGraphsNextIO.jl`

- responsibility: MetaGraphsNext native graph projection for library-created
  and caller-owned supplied targets
- interface: honest target validation, honest supplied-instance contract, and
  parity across package-owned and retained wrapper surfaces
- tested: canonical-owner tests, public-surface parity tests, network-rejection
  tests, and follow-up later-edge contract tests

### `ext/MetaGraphsNextAbstractTreesIO.jl`

- responsibility: `AbstractTrees` traversal compatibility through a wrapper on
  MetaGraphsNext-backed loaded graphs
- interface: traversal without moving tree semantics into core
- tested: MetaGraphsNext `AbstractTrees` integration tests

### `ext/PhyloNetworksIO.jl`

- responsibility: PhyloNetworks native graph projection and supplied-target
  commit discipline
- interface: rooted-network and tree-compatible native loads through the
  package-owned public surface
- tested: canonical-owner tests, public-surface tests, annotation-path tests,
  rejection-path tests, and shipped examples

### Public docs and examples

- responsibility: tell the same public contract story as the live owner and
  tests
- interface: README, index docs, PhyloNetworks docs, and shipped examples
- tested: docs build, example runs, and targeted rejection-path assertions

  authoritative tables, and row-reference access depend on.
- MetaGraphsNext constrains what an honest library-created or supplied-instance
  `MetaGraph` story can be, including same-typed zeroing and in-place graph
  mutation semantics.
- AbstractTrees defines traversal compatibility that can live cleanly in a
  wrapper over MetaGraphsNext-backed assets.
- PhyloNetworks constrains rooted-network construction and demonstrates the kind
  of native target semantics that the package-owned typed owner must project
  into honestly.
