# Audit Report: typed package-owned load architecture and post-audit contract repair

Parent PRDs: `design/original-prd.md`; `.workflow-docs/202605040131_type-stable-parse/01_prd.md`; `.workflow-docs/202605070011--type-stable-parse--audit-fix/01_prd.md`
Date: 2026-05-08
Files in scope: 52 live files plus 2 referenced-but-missing design authorities

## Summary

The live repository is green on direct verification: `julia --project=test test/runtests.jl` passed with `1374` tests, `julia --project=docs docs/make.jl` passed, and the three shipped example scripts under `examples/src/` ran successfully. The main typed-owner architecture is structurally sound: `read_lineages(...)` is the live package-owned public surface, `FileIO.load(...)` is a compatibility wrapper, `BuilderDescriptor` now rejects erased handles at the public boundary, and the MetaGraphsNext library-created target tightening is honest.

This effort is not yet cleanly complete, though. One high-severity Tranche 3a lock is only partially closed in the live code and tests, and the user-declared comprehensive authority corpus is internally inconsistent enough that a fresh maintainer cannot tell which legacy scope items are still controlling without project-owner classification.

## Handoff integrity

The later workflow chain is internally consistent on the ratified package-owned public surface. The tranche-3 decision record, `README.md`, `docs/src/index.md`, `docs/src/phylonetworks.md`, the public-surface tests, and the shipped examples all center `read_lineages(...)` as the package-owned public surface and classify `FileIO.load(...)` as compatibility-only.

Two handoff drifts remain:

- the active authority set still names missing legacy brief files and still contains older `load(...)`-centric contract language without an explicit supersession note
- the Tranche 3a remediation tasking claims owner-level proof of same-target retry and caller-owned graph integrity for later custom-edge-constructor throws that the current tests do not actually provide

## High findings

### 1. Tranche 3a caller-owned graph-integrity lock is only partially closed

**Location**: `.workflow-docs/202605070011--type-stable-parse--audit-fix/03_tranche-3a--remediation-tasking.md:537-543`; [test/extensions/metagraphsnext_canonical_owner.jl](/home/jeetsukumaran/site/storage/local/computing/research/20240812_Lineages.jl/LineagesWorkspace/LineagesIO.jl/test/extensions/metagraphsnext_canonical_owner.jl:483); [ext/MetaGraphsNextIO.jl](/home/jeetsukumaran/site/storage/local/computing/research/20240812_Lineages.jl/LineagesWorkspace/LineagesIO.jl/ext/MetaGraphsNextIO.jl:417); [ext/MetaGraphsNextIO.jl](/home/jeetsukumaran/site/storage/local/computing/research/20240812_Lineages.jl/LineagesWorkspace/LineagesIO.jl/ext/MetaGraphsNextIO.jl:170)
**Category**: Logic / Architecture / Handoff integrity
**Problem**: The Tranche 3a remediation tasking says the owner-level proof must show that failed supplied-instance custom-edge construction leaves caller-owned graphs unchanged and retryable, including a later rooted-network edge whose custom `EdgeData` constructor throws. The live owner-level test does not prove that. Instead, it codifies retained partial graph state after the throwing constructor (`nv == 4`, `ne == 3`) and performs no same-target retry. Because `validate_empty_metagraph` still rejects any non-empty supplied graph before binding starts, retrying that same object after the later-edge throw would now fail at the empty-instance guard rather than reproducing the original constructor failure. The implementation repair in `ext/MetaGraphsNextIO.jl` pre-materializes per-call edge payloads before child insertion, which honestly fixes the first-failing-edge dirty-graph bug, but it does not provide the whole-load retry or rollback guarantee that the Tranche 3a tasking claims is closed.
**Suggestion**: Reclassify this explicitly with the project owner. Either narrow the Tranche 3a contract and workflow text to the guarantee the code actually provides, or implement the stronger whole-load retry guarantee the tasking claims, which will require an owner-level rollback or copy-on-write story for caller-owned supplied graphs plus direct retry regressions for the later-edge-throw path.

**Location**: [ext/PhyloNetworksIO.jl](/home/jeetsukumaran/site/storage/local/computing/research/20240812_Lineages.jl/LineagesWorkspace/LineagesIO.jl/ext/PhyloNetworksIO.jl:194)
**Category**: Consistency / Best practices
**Problem**: The live PhyloNetworks rejection text still says the extension supports `load(src, HybridNetwork)` for library-created materialization and tells callers to use `load(src, target)` for supplied-target binding. That conflicts with the later ratified surface decision and the live README or docs, which classify `read_lineages(...)` as the package-owned public surface and `FileIO.load(...)` as compatibility-only. The mismatch is narrow, but it survives on a user-visible error path and is not directly guarded by a rejection-path test.
**Suggestion**: Rewrite this message to name `read_lineages(src, HybridNetwork)` as the first-class package-owned surface, mention the retained `load(...)` path as compatibility-only if desired, and add a focused rejection-path regression so the message does not drift again.
